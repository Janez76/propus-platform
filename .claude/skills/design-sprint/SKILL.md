description: Facilitate a Google Ventures-style Design Sprint (the 5-phase Map → Sketch → Decide → Prototype → Test process, or its compressed 1–2 day variant) to go from a fuzzy problem to a tested solution fast. Use when the user wants to validate a new feature/product direction, is stuck on "what should we even build", or asks to "run a design sprint" / "rapidly prototype and test an idea". Produces sprint artifacts, not production code.
---

You facilitate a Design Sprint (Knapp/GV method). Adapt the classic 5 days to whatever time the user has — even a single session — but keep the *sequence* and the *decision discipline*. Your job is to drive the process: ask the focusing questions, capture outputs, force decisions, prevent endless discussion.

## Before you start

- Get the **Sprint Question / challenge** in one sentence. Too big → narrow it.
- Identify who the **Decider** is (if it's the user, say so explicitly — they break ties).
- Confirm the **target customer** and the **key event/moment** you'll prototype and test.

## Phase 1 — Map (understand the problem)

- Write the **long-term goal** ("Why are we doing this? Where do we want to be in 1–2 years?").
- List **Sprint Questions** ("Can we…? Will users…? What has to be true?") — these become test criteria.
- Draw a simple **map**: customer(s) on the left, goal on the right, 5–15 steps of the journey between. Keep it minimal.
- Capture **"How Might We"** notes from any expert input — reframe problems as opportunities, then group and vote on them, place winners on the map.
- **Decider picks a target**: one customer + one moment on the map. That's the sprint's focus. Everything downstream serves it.

## Phase 2 — Sketch (generate solutions)

- **Lightning demos:** 15–20 min collecting good existing solutions (from anywhere — competitors, other industries). Capture the useful bit of each as a quick diagram.
- Four-step sketch, individually (no group brainstorm — solo work produces better, more varied ideas):
  1. **Notes** — walk the room, jot down what matters.
  2. **Ideas** — rough doodles, fill a page, no judgment.
  3. **Crazy 8s** — fold a sheet into 8, sketch 8 variations of one strong idea in 8 minutes. Forces alternatives.
  4. **Solution sketch** — one self-explanatory, anonymous, 3-panel storyboard of the best concept. Give it a catchy title. Detailed enough to stand alone.
- (When facilitating solo with the user: produce 3–5 distinct solution-sketch concepts yourself, described clearly, anonymized as "Concept A/B/C".)

## Phase 3 — Decide (choose without endless debate)

- **Art museum:** lay out all sketches.
- **Heat map:** everyone silently dot-votes the compelling parts. No talking.
- **Speed critique:** ~3 min per sketch — narrator summarizes, group calls out standout ideas, concerns noted, sketcher reveals only at the end. Strict timebox.
- **Straw poll:** each person picks one favorite, one dot, brief why.
- **Supervote:** the **Decider** places the deciding votes. That wins. No re-litigating.
- If two ideas both win and conflict → "Rumble" (prototype both, test head-to-head). If they're compatible → merge into one storyboard.
- Build the **storyboard:** ~10–15 panels, start one or two steps *before* the customer hits your product, then walk the whole tested experience end to end. Decide the level of fidelity. No new ideas here — only arrangement.

## Phase 4 — Prototype (build a façade, fast)

- **Goldilocks fidelity:** real enough to provoke honest reactions, fake enough to build in hours. It's a façade, not a foundation — throwaway.
- Pick the right tool (Keynote/Figma/clickable mock, a doctored real app, paper, a Wizard-of-Oz human-behind-the-curtain). 
- Assign roles: Makers, Stitcher (assembles into one coherent flow), Writer (real, specific copy — fake copy ruins tests), Asset Collector, Interviewer.
- **Trial run** at the end — walk the prototype, fix the obvious breaks.
- Write the **interview script** (see next phase).

## Phase 5 — Test (5 users, 1-on-1)

- **Five users** is enough — exposes ~85% of problems. One at a time, ~30–60 min each, observed by the rest of the team (separate room / screen share, taking notes).
- **Five-act interview:**
  1. Friendly welcome — set them at ease, "there are no wrong answers, think aloud, this isn't a test of *you*."
  2. Context questions — background, their relationship to the problem area.
  3. Introduce the prototype — "some things may not work, that's fine; I didn't build it so be honest."
  4. Tasks — give them a goal, not instructions; watch where they get stuck; ask "what is this? what do you expect? what would you do?" — open questions, no leading.
  5. Debrief — overall impressions, compare to alternatives, "magic wand" question.
- **Capture:** a grid — interviewees as columns, things-you're-watching-for as rows; mark positive/negative/neutral as you observe. After all 5, find the patterns (≥3 users hitting the same thing = real).
- **Conclude:** which Sprint Questions got answered? Decide: ship it / iterate the prototype / pivot / kill it. Name the next step explicitly.

## Compressed variants

- **2-day sprint:** Day 1 = Map + Sketch + Decide + storyboard; Day 2 = Prototype (morning) + Test (afternoon, fewer users ok).
- **Single session:** rapidly do Map → 3 concept sketches → Decider picks → describe the prototype + interview script → recommend running the 5 tests offline. Be honest that compression trades rigor for speed.

## Deliverable

The sprint artifacts: the map, the HMW winners, the solution concepts, the supervote outcome, the storyboard, a prototype spec, the interview script, and (after testing) the results grid + recommendation. Keep each artifact tight. This skill does not output production code — it outputs a validated (or invalidated) direction.
